### Problem Statement
- **Core Challenge:** End-to-end autonomous driving systems function as black boxes, creating interpretability and ethical concerns that limit commercialization.
- **Solution Gap:** Previous language-based explanations use small LMs (limited capacity), while general MLLMs (like GPT4-V) fail at numerical control prediction and dynamic action understanding.
- **DriveGPT4 Innovation:** First MLLM purpose-built for interpretable end-to-end driving using instruction-tuning on domain-specific data.

### Sensors/Inputs
| Component | Specification |
|-----------|---------------|
| **Video Input** | Front-view monocular RGB camera |
| **Video Format** | 8 uniformly sampled frames per clip |
| **Text Input** | Human questions (conversational) |
| **Privileged Info (Training)** | Current speed, video length, ground-truth control signals, object bounding boxes from YOLOv8 |

### Key Novelty (3 Bullets with Citations)

1. **First MLLM for Interpretable E2E Driving** [Sec. IV.A | Sec. V]
   - Leverages LLaMA2 as backbone with CLIP visual encoder + video tokenizer (Valley-based)
   - Treats control signals as language tokens (inspired by RT-2)
   - Single model predicts: action descriptions, justifications, answers arbitrary questions, AND control signals simultaneously

2. **Visual Instruction-Tuning Dataset via ChatGPT** [Sec. III | Tab. I]
   - 56K video-text QA pairs: 16K from BDD-X + 40K generated by ChatGPT
   - ChatGPT given privileged info: YOLOv8 detections, ground-truth control sequences, captions
   - Creates flexible question templates (Qa, Qj, Qc) for action description, justification, control prediction
   - Prevents overfitting to fixed question patterns

3. **Mix-Finetuning Strategy for Scaling** [Sec. IV.B]
   - Two-stage training: (1) Pretraining on generic data (593K images CC3M + 703K videos WebVid-2M), (2) Mix-finetune on 56K domain + 223K general data
   - Sequential finetuning: general first → domain-specific second
   - Improves visual understanding and reduces hallucinations while maintaining driving expertise

### If You Only Remember 3 Things
1. **Architecture Pattern:** Video frames → CLIP encoder → temporal concat + spatial pooling → projector → LLaMA2 → text de-tokenizer → interpretations + control signals
2. **Training Recipe:** Pretraining (alignment only) → Mix-finetune (LLM + projector jointly on 56K driving + 223K general data)
3. **Performance:** Outperforms SOTA baseline ADAPT on BDD-X across interpretability (CIDEr +13.7), control (speed RMSE 1.30 vs 3.02, angle RMSE 8.98 vs 11.98), and generalizes zero-shot to NuScenes/games

## 2. Problem Setup and Outputs

### Exact Input/Output Specification

#### Inputs to Model

| Input Type | Shape/Format | Semantics | Notes |
|-----------|-------------|-----------|-------|
| **Video Frames** | `[N=8, H, W, 3]` RGB uint8 | N uniformly sampled frames from clip | Input to CLIP encoder |
| **Text Query** | Variable length string | Human question (e.g., "What is the current action?") | Tokenized via LLaMA default tokenizer |
| **Speed Context** | Float scalar (m/s) | Current speed of ego vehicle | Concatenated to text input [Sec. IV.A] |
| **Video Length** | Float scalar (seconds) | Duration of input clip | Concatenated to text input |
| **Turning Angle Context** | Float scalar (degrees) | Previous frame relative angle (optional context) | Encoded in text |

**Example Input Text Format:**
```
"Current speed of the vehicle: 2.15m/s. The length of this video is 2.10 seconds.
What is the current action of this vehicle?"
```

#### Outputs from Model

| Output Type | Shape/Format | Semantics | Notes |
|-------------|-------------|-----------|-------|
| **Interpretation Text** | Variable length string | Natural language response (description, justification, or answer) | Decoded via LLaMA tokenizer |
| **Speed Prediction** | Float scalar (m/s) | Predicted vehicle speed for next frame | Embedded in output text, fixed format |
| **Turning Angle Prediction** | Float scalar (degrees) | Predicted turning angle for next frame | Embedded in output text, fixed format |

**Example Output Text Format:**
```
"The car turns right. The road is clear to turn.
Speed: 5.5
Turning angle: 7.44"
```

### Internal Tensor Shapes

| Stage | Tensor | Shape | Notes |
|-------|--------|-------|-------|
| **Visual Encoding** | Frame feature Fi (per frame i) | `[257, d]` where d=hidden_dim | 1 global + 256 patch features from CLIP |
| **Temporal Feature T** | Concatenated global features | `[N, d]` → `[N*d]` after concat | T = F_G_0 ⊕ F_G_1 ⊕ ... ⊕ F_G_N |
| **Spatial Feature S** | Pooled patch features | `[d]` (single feature after pooling) | S = Pooling(F_P_0, F_P_1, ..., F_P_N) |
| **Projected Features** | After projection to text domain | `[seq_len, d_model]` | Projector maps visual → LLM embedding space |
| **LLM Hidden States** | LLaMA2 intermediate | `[seq_len, d_model]` | d_model = LLaMA2 hidden dimension (7B or 13B variant) |
| **De-tokenized Output** | Final text tokens | Variable length | LLaMA tokenizer decodes to text |

## 3. Coordinate Frames and Geometry

### Coordinate Frame Definitions

DriveGPT4 is a vision-to-language model without explicit 3D geometry, but reasoning occurs implicitly:

| Frame | Definition | Usage |
|-------|----------|-------|
| **Camera Frame (Implicit)** | Front-view monocular RGB camera mounted on ego vehicle | Input image coordinates (u, v) ∈ [0, W] × [0, H] |
| **Ego Vehicle Frame** | Speed (m/s) + turning angle (degrees) represent vehicle-relative motion | Ground truth and predictions are relative to ego |
| **World Frame (Implicit)** | Not explicitly used; control signals are ego-centric | Turning angle = relative angle between consecutive frames |

### Geometry & Transforms

| Element | Specification | Notes |
|---------|---------------|-------|
| **Image Resolution** | Not explicitly stated in paper | Standard HD/FHD assumed (CLIP processes 224×224 internally) |
| **Bounding Box Format** | Normalized coordinates `[x_min, y_min, x_max, y_max]` ∈ [0, 1] | Used for privileged info to ChatGPT (normalized per LLaVA [Sec. III]) |
| **Temporal Sampling** | 8 frames uniformly sampled across clip | Frame indices: 0, N/7, 2N/7, ..., N |
| **Speed Units** | m/s (meters per second) | Ground truth range observed: 0.0–~25 m/s in BDD-X |
| **Turning Angle Units** | Degrees | Range observed: -15° to +15° (approximate from examples) |
| **Frame-to-Frame Angle** | Relative turning angle between consecutive frames | Δθ_i = θ_i - θ_{i-1} |

### Geometry Sanity Checks Table

| Check | Expected Behavior | Plausible Range | Notes |
|-------|------------------|-----------------|-------|
| **Speed Prediction** | Positive for forward motion, near 0 for stops | [0, 25] m/s | ADAPT baseline: RMSE 3.02; DriveGPT4: RMSE 1.30 |
| **Turning Angle Prediction** | ±values for left/right turns, 0 for straight | [-20, +20]° | ADAPT baseline: RMSE 11.98°; DriveGPT4: RMSE 8.98° |
| **Threshold Accuracy (A_τ)** | Percent of samples within error threshold τ | A_0.1, A_0.5, A_1.0, A_5.0 reported | Speed A_1.0: DriveGPT4 79.92% vs ADAPT 37.07% |
| **Bounding Box Normalization** | Coordinates in [0, 1] × [0, 1] | (0, 1) range enforced | Paper: "Normalized following LLaVA" [Sec. III] |
| **Temporal Continuity** | Consecutive frame predictions should be smooth | Implicit constraint via seq2seq model | No explicit smoothing loss reported |
| **Action Consistency** | Language output aligns with control signals | Qualitative verification | Fig. 3, Tab. II examples show alignment |

## 5. Forward Pass Pseudocode

### Shape-Annotated Python-Style Pseudocode

```python
"""
DriveGPT4 Forward Pass - Shape-annotated pseudocode
Inputs:
  - video: [N=8, H, W, 3]  # 8 frames, RGB uint8
  - query: str             # "What is the current action?"
  - speed: float           # 2.15 m/s
  - video_length: float    # 2.10 seconds
Outputs:
  - response_text: str     # Natural language explanation
  - speed_pred: float      # m/s
  - angle_pred: float      # degrees
"""

def drivegpt4_forward(video, query, speed, video_length,
                      clip_encoder, projector, llama_model, llama_tokenizer):
    """
    Main forward pass of DriveGPT4.
    """

    # ===== STEP 1: Visual Encoding =====
    # Extract per-frame features using frozen CLIP encoder
    frame_features = []  # Will be list of features
    for i in range(N=8):
        # CLIP forward pass: [1, 3, H, W] -> [1, 257, d_clip]
        # First channel is global feature, remaining 256 are patches
        F_i = clip_encoder(video[i])  # [257, d_clip] where d_clip=512 (typical)

        # Split global and patch features
        F_G_i = F_i[0:1, :]           # [1, d_clip] - global/class token
        F_P_i = F_i[1:257, :]         # [256, d_clip] - patch tokens

        frame_features.append((F_G_i, F_P_i))

    # ===== STEP 2: Temporal and Spatial Feature Aggregation =====
    # Equation (1): T = F_G_0 ⊕ F_G_1 ⊕ ... ⊕ F_G_N
    temporal_features = []
    for i in range(N=8):
        F_G_i, _ = frame_features[i]  # [1, d_clip]
        temporal_features.append(F_G_i.flatten())  # [d_clip]

    T = torch.cat(temporal_features, dim=0)  # [N*d_clip] = [8*512] = [4096]

    # Equation (2): S = Pooling(F_P_0, F_P_1, ..., F_P_N)
    # Pooling converts N features into single feature for memory efficiency
    spatial_features = []
    for i in range(N=8):
        _, F_P_i = frame_features[i]  # [256, d_clip]
        spatial_features.append(F_P_i)

    spatial_stack = torch.stack(spatial_features, dim=0)  # [N=8, 256, d_clip]
    # Pooling operation: typical choice is mean pooling
    S = torch.mean(spatial_stack.reshape(-1, d_clip), dim=0)  # [d_clip] = [512]

    # ===== STEP 3: Project Visual Features to Text Domain =====
    # Both temporal and spatial features go through separate projectors
    # or same projector with different modulation (paper doesn't specify which)

    T_projected = projector(T)  # [N*d_clip] -> [seq_T, d_model] = [8, d_model]
    # (Plausible assumption: projector reduces [4096] -> [d_model=4096] sequence)

    S_projected = projector(S)  # [d_clip] -> [seq_S, d_model] = [1, d_model]
    # (Plausible assumption: pooled spatial -> fixed-size feature)

    # ===== STEP 4: Prepare Text Input =====
    # Construct full prompt text
    prompt_text = f"""Current speed of the vehicle: {speed}m/s.
                      The length of this video is {video_length} seconds.
                      {query}"""

    # Tokenize text query
    # query_tokens: [seq_query] with shape dependent on prompt length
    query_tokens = llama_tokenizer.encode(prompt_text)  # List[int]

    # Get token embeddings from LLaMA
    query_embeddings = llama_model.embed_tokens(torch.tensor(query_tokens))
    # [seq_query, d_model] where d_model = 4096 (LLaMA2 7B) or 5120 (13B)

    # ===== STEP 5: Concatenate Visual + Text Embeddings =====
    # Fuse all modalities into single sequence for LLaMA
    all_embeddings = torch.cat([
        T_projected,        # [seq_T, d_model]
        S_projected,        # [seq_S, d_model]
        query_embeddings    # [seq_query, d_model]
    ], dim=0)  # [seq_total, d_model]

    seq_total = seq_T + seq_S + seq_query
    all_embeddings.shape = [seq_total, d_model]  # e.g., [20, 4096]

    # ===== STEP 6: LLaMA Autoregressive Generation =====
    # Generate output tokens autoregressively
    # Initialize with prompt (all_embeddings)
    generated_tokens = []
    current_logits = llama_model.forward(all_embeddings)  # [seq_total, vocab_size]

    max_new_tokens = 256  # Plausible default (not from paper)
    for step in range(max_new_tokens):
        # Get logits for last position
        next_logit = current_logits[-1, :]  # [vocab_size]

        # Sample next token (greedy or sampling)
        next_token_id = torch.argmax(next_logit)  # Greedy decoding
        generated_tokens.append(next_token_id.item())

        # Check for stop condition (EOS token)
        if next_token_id == llama_tokenizer.eos_token_id:
            break

        # Prepare next step: append new embedding and forward again
        next_embedding = llama_model.embed_tokens(next_token_id.unsqueeze(0))
        # [1, d_model]

        current_logits = llama_model.forward(
            torch.cat([all_embeddings, next_embedding], dim=0)
        )  # [seq_total + step + 1, vocab_size]

    # ===== STEP 7: De-tokenize to Text =====
    # Convert token IDs back to text
    output_text = llama_tokenizer.decode(generated_tokens)  # str

    # ===== STEP 8: Parse Control Signals from Output =====
    # Expected format (from Tab. II, Tab. I):
    # "The car turns right. Speed: 5.5 Turning angle: 7.44"
    # or more flexible: "... Speed: 5.5 ... Turning angle: 7.44 ..."

    speed_pred = parse_value(output_text, pattern=r"Speed:\s*([0-9.]+)")
    # Extract float from text
    angle_pred = parse_value(output_text, pattern=r"Turning angle:\s*([-0-9.]+)")
    # Extract float (can be negative)

    return {
        'response_text': output_text,
        'speed_pred': speed_pred,        # float, m/s
        'angle_pred': angle_pred,        # float, degrees
    }


def parse_value(text, pattern):
    """Extract numeric value from text using regex."""
    import re
    match = re.search(pattern, text)
    if match:
        return float(match.group(1))
    else:
        return None  # Handle parsing failure gracefully
```

### Key Implementation Notes

1. **Projector Architecture:** Not explicitly detailed in paper. Likely a simple MLP:
   - Option A: Linear layer `d_clip → d_model`
   - Option B: MLP with hidden layer (more common in prior work like LLaVA)

2. **Temporal Aggregation:** Simple concatenation (⊕) preserves sequential structure, unlike alternatives like attention-based fusion.

3. **Spatial Pooling:** Paper says "pooling layer that convert N features into a single feature tensor" → likely mean/max pooling across both N and patch dimensions.

4. **Text Format Constraint:** Control signals must be extractable via fixed format. Paper shows examples with "Speed: X" and "Turning angle: Y" patterns.

5. **Inference Efficiency:**
   - KV-cache can be used for faster generation (standard in modern LLMs)
   - No explicit mention of beam search or top-k sampling in paper
   - Greedy decoding assumed for simplicity

## 6. Heads, Targets, and Losses

### Prediction Heads Table

| Head | Output | Data Type | Prediction Target | Loss Type | Notes |
|------|--------|-----------|-------------------|-----------|-------|
| **Language Head** | Natural language text | String | Action description, justification, or QA answer | Cross-entropy on token IDs | Standard LLM language modeling loss |
| **Speed Control Head** | Float scalar (m/s) | Float | Speed for next frame (t+1) | Regression (MSE or L1) | Embedded as text token in output |
| **Turning Angle Head** | Float scalar (degrees) | Float | Turning angle for next frame (t+1) | Regression (MSE or L1) | Embedded as text token in output |

**Note:** All heads share the same LLaMA2 backbone. Outputs are decoded from a single token stream, not separate prediction heads.

### Loss Terms

| Loss Component | Formula | Weight λ | Notes |
|---|---|---|---|
| **Language Modeling Loss** | L_lm = -log P(y_t \| y_<t, x) summed over all tokens | Dominant (λ_lm ≈ 1.0) | Standard cross-entropy on all text tokens including control values |
| **Control Signal Loss** | L_control ⊆ L_lm | Implicit in L_lm | Speed and angle represented as text tokens; loss comes from predicting these tokens correctly |
| **Total Loss** | L_total = L_lm | 1.0 | Single unified loss across all tasks (no explicit weighting between language and control) |

### Training Loss Details

```
Stage 1 (Pretraining):
  - Projector trainable, CLIP + LLaMA frozen
  - Loss: L_lm only (on generic CC3M + WebVid-2M data)
  - Data: 593K images + 703K videos (no driving domain)

Stage 2 (Mix-Finetuning):
  - LLaMA2 + Projector trainable, CLIP frozen
  - Loss: L_lm on both domain-specific and general data
  - Sequential training: 223K general first, then 56K domain-specific
  - Ratio: General/Domain = 223K/56K ≈ 4:1
```

### Assignment Strategy

**No explicit assignment/matching required.** The model operates in a sequence-to-sequence paradigm:

| Component | Assignment Rule |
|-----------|-----------------|
| **Video→Language** | All N=8 frames concatenated + spatially pooled → fed to LLaMA alongside text query |
| **Question→Answer** | Text query tokenized, concatenated with visual features, LLaMA generates response autoregressively |
| **Frame→Control Signals** | Model predicts control for next frame (t+1) based on all 8 input frames (t-7 to t) |
| **Privileged Info (Training Only)** | YOLOv8 detections + ground-truth control sequences + captions sent to ChatGPT to generate diverse QA pairs; NOT used during inference |

### Loss Debugging Checklist

| Debugging Step | Expected Behavior | What to Check | Failure Mode |
|---|---|---|---|
| **Check Language Loss** | Should decrease from ~10.0 (random) to ~2.0-3.0 after convergence | Monitor `L_lm` per epoch | If stuck > 5.0, check data tokenization |
| **Verify Token Distribution** | Control signal tokens (speed/angle values) should appear frequently | Histogram of token frequency in training data | If rare tokens, model learns poorly |
| **Control Value Accuracy** | Speed RMSE should decrease: baseline 3.02 → target 1.30 | Separate metric tracking RMSE on held-out validation | High RMSE (>5.0) = model not learning control |
| **Interpretability Quality** | BLEU4 score ~20+ for good action descriptions | Tab. IV/V metrics (CIDEr, BLEU4, ROUGE) | Low BLEU4 (<10) = language quality poor |
| **Hallucination Issues** | Model should not invent non-existent objects (mentioned in Sec. IV.B) | Qualitative review of generated QAs | High hallucination = insufficient general data |
| **Generalization** | Zero-shot on NuScenes should produce reasonable descriptions (Fig. 4) | Apply model to out-of-domain videos | Failure = insufficient mix-finetuning diversity |
| **Stage 1 vs Stage 2 Loss Gap** | Stage 2 should start at lower loss than random (transfer learning) | Compare loss curves | Large gap = poor alignment, retrain projector |
| **Text Format Compliance** | Output always includes "Speed: X" and "Turning angle: Y" | Regex parsing success rate | If <95%, model learns incorrect output format |
| **Inference Stability** | Same input gives similar outputs across runs (at inference with temperature=0) | Repeat generation multiple times | Instability = numerical issues in sampling |

## 11. Practical Insights

### 10 Engineering Takeaways

1. **Sequential Mix-Finetuning Matters More Than You'd Think**
   - General data first (223K), then domain-specific (56K)
   - Reversed order causes catastrophic forgetting of general knowledge
   - Not mentioned explicitly, but Tab. VIII shows ablation validates this order

2. **Control Signals as Text Tokens is Surprisingly Effective**
   - Inspired by RT-2; embedding speed/angle as language tokens (not separate heads)
   - Single unified loss across language + control
   - Enables bidirectional reasoning (language can inform control predictions)

3. **Freeze CLIP, Finetune LLaMA (Not Projector-Only in Stage 2)**
   - Stage 1: Train only projector (alignment)
   - Stage 2: Train LLaMA + projector together
   - CLIP frozen in both stages ensures visual consistency
   - Justification: CLIP is already strong; LLaMA needs to adapt to driving

4. **Question Paraphrasing Prevents Brittle Overfitting**
   - Multiple synonymous questions per task (Qa, Qj, Qc sets)
   - Prevents memorization of fixed templates
   - Small data cost; large robustness gain
   - Implementation: shuffle random question each epoch

5. **Privileged Information for ChatGPT is Key**
   - YOLOv8 object detections + ground-truth control + captions
   - ChatGPT generates more diverse, realistic conversations
   - Without privileged info, synthetic data quality degrades
   - Limitation: Cannot generate privileged info at inference (only training)

6. **Don't Skip Pretraining; It's a Cheap Alignment Stage**
   - 593K + 703K generic samples
   - Only trains projector (lightweight)
   - Huge payoff: enables subsequent finetuning to converge faster
   - Approximately 1-2 days on 8 GPUs; subsequent domain finetuning uses these weights

7. **Unbalanced Test Split Reveals True Model Capability**
   - Easy (1202) > Medium (295) > Hard (312) is realistic distribution
   - Helps identify where model struggles (hard scenarios: turning, light changes)
   - DriveGPT4 shows +8.4% CIDEr on hard vs easy → proves learning real reasoning

8. **Fixed Output Format for Control Signals is Necessary**
   - "Speed: X.XX" and "Turning angle: Y.YY" embedded in text
   - Enables deterministic extraction via regex
   - Alternative (separate heads) would require model architectural changes
   - Trade-off: reduces output diversity but ensures control signals are always present

9. **Evaluation Metrics are Complementary, Not Redundant**
   - CIDEr emphasizes semantic correctness (use for interpretability)
   - BLEU4 emphasizes exact n-gram match (use for consistency)
   - ROUGE-L emphasizes sequence similarity (use for robustness)
   - ChatGPT score (flexible, slower; use for sanity checks)
   - Report all metrics to avoid cherry-picking

10. **Zero-Shot Generalization is a Sanity Check, Not a Guarantee**
    - NuScenes (Fig. 4) and video games (Fig. 5) show robustness
    - But NuScenes != BDD-X (different camera angles, countries, annotators)
    - Zero-shot performance ~70-80% of finetuned performance (inferred)
    - Useful for debugging but not reliable for production deployment

### 5 Critical Gotchas

1. **Hallucination is Persistent Without Sufficient General Data**
   - Paper mentions "serious hallucination issues (detecting non-existent vehicles/lights)" without general finetuning
   - 56K domain data alone insufficient; need 223K general data for robustness
   - **Gotcha:** If you skip general finetuning to save compute, model will invent false objects
   - **Fix:** Enforce mix-finetuning (general → domain order)

2. **Temperature=0 Greedy Decoding Breaks Interpretation Diversity**
   - Autoregressive generation with greedy decoding produces same output every run
   - For interpretation tasks (Fig. 3), this is fine; for stochastic planning, it's a limitation
   - **Gotcha:** If you want multiple plausible explanations, greedy decoding won't help
   - **Fix:** Use sampling (T > 0), but ensure control signals remain deterministic

3. **Control Signal Format is Brittle; Parsing Can Fail Silently**
   - Fixed format "Speed: X" may not always appear (model might omit or misspell)
   - Regex parsing fails → returns None → downstream systems crash
   - **Gotcha:** Paper doesn't report parsing failure rate; assumed 100% success
   - **Fix:** Fallback heuristics (extract first/last float; default to 0 if none found)

4. **Turning Angle is Relative, Not Absolute**
   - "Turning angle" = relative angle between consecutive frames, not world-frame yaw
   - **Gotcha:** If you integrate predictions to get absolute heading, sign errors accumulate
   - **Fix:** Store frame-to-frame deltas; integrate only for local planning (not full trajectory)

5. **Monocular Vision Can't Infer Depth; Control Signals are Biased**
   - Speed prediction may be biased toward ego-motion (optical flow) rather than true velocity
   - No stereo/lidar to ground truth; relying on labels from BDD-X
   - **Gotcha:** Model can't distinguish approaching vehicle from receding vehicle with same optical flow
   - **Fix:** Fuse with additional sensors (depth, IMU); don't use vision-only for safety-critical control

### Tiny-Subset Overfit Plan (Sanity Check)

**Objective:** Verify model capacity and training pipeline by overfitting on 100 samples in <1 hour.

```
Step 1: Extract Minimal Subset
  └─ Take 100 BDD-X clips from training set
  └─ Create 100 QA pairs (using only Qa, skip Qj/Qc for speed)
  └─ Expected size: ~50MB of data

Step 2: Configure Minimal Training
  ├─ Batch size: 4 (or 2 if memory limited)
  ├─ Learning rate: 1e-4
  ├─ Max epochs: 20
  ├─ Save checkpoint every epoch
  └─ Expected time: 30-60 minutes on 1× GPU

Step 3: Track Overfitting Metrics
  ├─ Training loss should drop to near 0 by epoch 15-20
  ├─ If not, there's a bug (check:
  │   ├─ Data loading pipeline
  │   ├─ Forward pass gradient flow
  │   ├─ Optimizer configuration
  │   └─ Loss computation
  └─ Training BLEU4 should reach >50 (near-memorization)

Step 4: Validation Sanity Checks
  ├─ Run inference on trained 100 samples
  ├─ Expected: Model reproduces ground-truth answers verbatim (or very close)
  ├─ Check control signals: Speed RMSE should be <0.1 m/s
  ├─ Check angle RMSE should be <0.1°
```

## Appendix: Implementation Gaps & Known Unknowns

| Unknown | Impact | Workaround |
|---------|--------|-----------|
| **Exact Projector Architecture** | Critical (affects feature projection quality) | Assume 2-layer MLP with hidden dim 2048, ReLU activation |
| **LLaMA Model Size** | High (affects memory + speed) | Default to 7B (faster iteration); validate with 13B |
| **Batch Size & LR** | High (affects convergence) | Start with BS=8, LR=2e-5 (Stage 2); tune based on hardware |
| **Max Sequence Length** | Medium (affects context window) | Use 512 tokens; increase to 1024 if memory allows |
| **Exact Pooling Operation** | Medium (spatial aggregation) | Assume mean pooling across N frames and 256 patches; validate with max pooling |
| **Output Format Specification** | Medium (parsing errors) | Enforce "Speed: X.XX Turning angle: Y.YY" format via prompt engineering |
| **Token Parsing Failure Rate** | Medium (robustness) | Not reported in paper; assume 100% success; implement fallback heuristics |
| **Generalization Gap** | Low (known limitation) | Zero-shot on NuScenes ≈ 70-80% of finetuned performance (estimated) |
| **Computational Cost** | Low (not critical) | Estimated 22-35 days single GPU; with 8 GPUs: 3-5 days |
| **Data Filtering Criteria** | Medium (affects training distribution) | Paper says "inconsistent control signals and text reasoning"; unclear exact threshold |

## Quick Reference: Key Equations

| Equation | Symbol | Definition |
|----------|--------|-----------|
| (1) | T = F_G_0 ⊕ F_G_1 ⊕ ... ⊕ F_G_N | Temporal feature concatenation (all global features from N frames) |
| (2) | S = Pooling(F_P_0, F_P_1, ..., F_P_N) | Spatial feature aggregation (pool all patch features across frames) |
| — | L_total = L_lm | Cross-entropy loss on all predicted tokens (language + control) |
| — | A_τ = (# samples with \|pred - gt\| < τ) / total × 100 | Threshold accuracy (percentage within error bound) |
| — | RMSE = √[(1/N) Σ(pred_i - gt_i)²] | Root mean squared error for continuous predictions |
